# Pitfalls — Phase 1.1.3 shape_dispatch execution (closed 2026-06-02)

Subagent-driven execution of the shape_dispatch 2×2 workload (6 waves, two-stage
review per task). Session was interrupted mid-Wave-2 by a usage limit and resumed.
Lessons below; P1/P2 concrete fixes already filed as tech-debt this session, P3
inline-applied to CLAUDE.md.

## Tooling

### Sanity-check a verification tool on a known-positive before trusting a zero/negative

**What happened.** R1 (devirtualization) verification ran `wasm-objdump -d <wasm> | grep -c
call_indirect` and expects `> 0` for `*_dyn` binaries. `wasm-objdump` (wabt) is not
installed on this machine; the invocation silently produced `0` for *every* binary —
which is exactly the "devirt fired" failure signal. A subagent caught it only by
sanity-checking the known-positive `mixed_dyn` reference (it should be non-zero, read 0 →
tool suspect) and switching to `.tools/wasi-sdk-25/bin/llvm-objdump`.

**Root cause.** A verification command whose absent-tool / empty-input path returns the
same value as the genuine failure signal. `grep -c` on empty input = `0` = "bad".

**Prevention.** When a check's failure signal is a specific value (0, empty, non-match),
first run it against a case you *know* should pass; if that reads as failure, the tool/
command is broken, not the artifact. Filed concrete fix: `docs/tech_debt/r1-devirt-objdump-tooling.md`
(resolve a present disassembler via `tool-paths.ts`; make absent-tool a hard error).

## Process

### Bench "0 failures" ≠ no correctness failures — scan result JSONs, don't trust exit code / failures.txt

**What happened.** The full bench run exited 0 and wrote no `failures.txt`, so the plan's
Task-25 check reported clean. But 12 result JSONs carried `quality.validated: false` /
`correctnessFailed: true` (hashmap_int cpp-emscripten lookup+delete at L). The same 12
were present — bit-identical — in the Phase 1.1.2.1 baseline, meaning the defect crossed
two phase closes undetected.

**Root cause.** The run orchestrator records process-level crashes in `failures.txt` /
exit code, but a per-case `correctnessFailed` does not flip the run-level success signal.
"0 failures" is satisfiable on top of silently-wrong data.

**Prevention.** Add an explicit post-run scan to close gates: `grep -rl '"validated": false'
<out>/*.json` must be empty. Don't equate "exit 0 + no failures.txt" with correctness.
Filed: `docs/tech_debt/bench-run-correctness-fail-not-surfaced.md`.

### Before reverting a subagent's apparent "convention divergence," verify it isn't load-bearing

**What happened.** A subagent's interrupted work left 3 JS `tsconfig.json` deviating from
the 6 peer benches (it dropped the `noUncheckedIndexedAccess: false` override). On resume
I reverted them "for peer consistency" — and broke `pnpm lint:ts`: under the loose setting
`buf[off]` is `number`, so the code's `as ShapeKind` cast became an *unnecessary* assertion
(`@typescript-eslint/no-unnecessary-type-assertion` error). The strict-tsconfig and the
cast were coupled: the divergence was deliberately solving a real lint conflict (enum-tagged
input parsing), not arbitrary drift.

**Root cause.** Treated a deviation as noise without checking *why* it was introduced.
The deviation was satisfying a gate; reverting it re-opened the gate failure.

**Prevention.** Before reverting an apparent convention-divergence, run the gate it might be
load-bearing for (lint/typecheck/test) on both the divergent and the "consistent" version.
If the consistent version fails a gate the divergence passed, the divergence is intentional —
keep it and document *why*. (Echoes memory `feedback-grep-before-scope`: understand the
existing text/state before overwriting it.) Inline-applied to `CLAUDE.md` § Tooling environment.
