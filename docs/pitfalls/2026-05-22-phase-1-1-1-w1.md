## 1. `scripts/build-rust.ts` читал stale wasm из per-crate target/

**What happened.** В Task 5 при добавлении нового wasm export'а `matmul` (alias для
`run`) в `benches/matmul/rust/raw/src/lib.rs` — после `pnpm build:rust` новый export
не появлялся в `dist/matmul/rust-raw-speed/module.wasm`. Cargo build выполнялся
успешно («Finished `release` profile in 0.00s»), но dist'у попадал **старый**
wasm. Источник: `scripts/build-rust.ts` копировал из
`benches/matmul/rust/raw/target/wasm32-unknown-unknown/release/matmul_rust_raw.wasm`
(stale, mtime May 5), вместо workspace-root `target/wasm32-unknown-unknown/release/...`
(где cargo пишет).

**Root cause.** Cargo workspace всегда пишет артефакты в `<workspace_root>/target/`,
независимо от `cwd` каждой `cargo build` команды. Скрипт делал `cwd: crateDir` →
`cargo build`, затем читал `join(crateDir, "target", ...)` — это per-crate target
из pre-workspace эры (когда крейт был standalone). Per-crate dir сохранил May-5
бинари; их не обновляли. Smoke не ломался, потому что Phase 1.1.0 W2 wasm
refactors (`addr_of!(HEAP.0)`, `with_slices`) были **byte-preserving** — старый
wasm функционально совпадал с новым, smoke его принимал.

**Prevention.**
- **Symptom rule.** Когда добавляешь **non-byte-preserving** change в Rust
  cargo crate и `pnpm build:rust` отрабатывает, но изменения не видны в
  `dist/.../module.wasm` — первое подозрение: пишутся ли артефакты в workspace
  target/, а не в crate-local target/?
- **Hard rule.** В scripts orchestrating `cargo build`: derive output path from
  workspace root, не cwd-relative. (Применено в `scripts/build-rust.ts:25-27`.)
- **CLAUDE.md gate.** Convention зафиксирована в § «Tooling environment» — теперь
  всегда в контексте при writing-plans для cargo build scripts.

## 2. tsx IPC socket требует sandbox bypass

**What happened.** В этой сессии многократно ловил `EPERM listen
/tmp/claude-501/tsx-501/*.pipe` на `pnpm smoke`, `pnpm exec tsx -e '...'`, и любых
`tsx`-launched subprocess. Pure `pnpm typecheck/lint/test` — работали в sandbox без
проблем (user confirmed: «Сейчас pnpm должен в sandbox работать тоже»).

**Root cause.** tsx использует Unix IPC socket для inter-process communication
(forking child workers). Создание socket в `/tmp/claude-501/tsx-501/*.pipe`
блокируется sandbox writable allowlist. `pnpm`-без-tsx (pure vitest/eslint/tsc)
не форкает таких subprocesses, поэтому работает.

Предыдущий pitfall #4 из `docs/pitfalls/2026-05-21-phase-1-1-0-execution.md`
описывал это как «pnpm всегда требует sandbox bypass» — это outdated:
сужение области до **tsx-launched subprocesses**, не всех `pnpm` команд.

**Prevention.**
- **Hard rule.** Любая `pnpm`-команда которая делегирует к `tsx scripts/<file>.ts`
  (smoke, build:*, fixtures, run-matrix, etc.) — `dangerouslyDisableSandbox: true`.
  Pure `pnpm typecheck/test/lint:ts` — sandbox OK.
- **CLAUDE.md gate.** Convention зафиксирована в § «Tooling environment».
- **Previous pitfall #4** теперь outdated; не обновляем (closed-phase file), новый
  pitfall более точный.

## 3. Plan executor: required-flag coupling между tasks

**What happened.** Plan разбивал `runner-node --entry` (Task 11) и `run-matrix
per-entry iteration` (Task 13) на отдельные tasks. Когда Task 11 landed
(required `--entry` в runner-node CLI), smoke падал немедленно потому что Task 13
(`run-matrix` передаёт `--entry`) ещё не сделан. Пришлось bundle'ить два task'а в
один коммит (`46eede7`) — intermediate state ломал smoke gate.

**Root cause.** При добавлении нового **required** CLI flag в downstream worker —
upstream callers должны передавать его атомарно. Plan не маркировал эту связь.
Альтернативный подход — сделать flag optional с дефолтом — не был выбран из-за
семантики (entry distinct from benchmarkId в long term).

**Prevention.**
- При writing-plans force decision: либо «все callers в одном коммите»
  (документировать coupling в plan'е), либо «flag optional с default = X» для
  intermediate gate green.
- Tracked в `docs/tech_debt/incorporate-pitfalls-2026-05-22.md` § Item 1 как
  candidate update для `superpowers:writing-plans` skill checklist.
